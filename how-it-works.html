Stage 1 — Smart Train Matching

Graph-powered route intelligence

Not a dumb table scan. Neo4j applies four filters simultaneously via Cypher traversal.

Line Detection

Auto-identifies WR, CR, or HR based on which stations appear in the search. No user input needed.

detect_line(origin, destination) → "WR"

Direction Inference

Compares station indices in the ordered line map. Borivali(6) → Churchgate(26) = southbound.

idx[origin] < idx[dest] → "southbound"

Time Window

Shows trains departing from 15 min before to 3 hours after search time. No 5am trains in an 8am search.

t - 15min ≤ departure ≤ t + 3hr

Route Coverage

Verifies the train covers your origin → destination segment. A Bandra-Churchgate train won't appear for Virar.

stops.includes(origin) && stops.includes(dest)
Stage 2 — Intelligence Engine

Three signals, weighted fusion

Each train is scored independently. No two trains get the same analysis — even on the same route.

50%

Historical Baseline

query: train_number + day_type + hour_bucket

Aggregates all crowd reports for this specific train, on this type of day (weekday/saturday/sunday), during this hour slot. RED = 100, YELLOW = 50, GREEN = 0.

Virar Fast 90015 | Monday | 08:00-09:00
56 reports: 40 RED, 12 YELLOW, 4 GREEN
score = (40/56 × 100) + (12/56 × 50) = 82
30%

Crowd Trend Detection

compare: last_7_days vs all_time_baseline

Detects if crowding is getting worse or better than usual. Catches surges from festivals, exams, strikes, or cascading delays.

Baseline RED% = 61% | This week RED% = 82%
Shift = +21% → trend = "spiking" → score = 85

Shift = -6% → trend = "stable" → score = 30
20%

Live Weather Multiplier

source: OpenWeatherMap API — real-time Mumbai

Rain drives more commuters to trains. Heavy rain causes platform rushing and slippery footboards. Acts as a crowd multiplier on top of the baseline.

Heavy rain (>10mm/hr) → score = 80
Light rain (2-10mm/hr) → score = 45
Clear / dry season → score = 0

Anti-Spam Threshold Consensus Protection

A single user can't flip a train's badge. The engine requires 2+ recent reports to agree before shifting from the computed baseline. This prevents trolling and ensures only genuine crowd shifts are reflected.

Stage 3 — Score Fusion

Weighted sum → single badge

Fusion formula
Historical × 0.50 + Trend × 0.30 + Weather × 0.20 = Final Score
Example: (82 × 0.50) + (85 × 0.30) + (0 × 0.20) = 41 + 25.5 + 0 = 67 → YELLOW
Adaptive: When weather is unavailable, weights shift to (hist × 0.60 + trend × 0.40) — no dilution
GREEN0 – 39 — Board safely
YELLOW40 – 69 — Expect crowds
RED70 – 100 — Skip this one
Stage 4 — LLM Reasoning

Natural-language explanation per train

The badge alone isn't enough. GPT-4o-mini generates a tone-matched one-liner that never contradicts the badge.

What the LLM receives

  • Badge: RED / YELLOW / GREEN
  • Train name and type (Fast / Slow)
  • Day type (weekday / saturday / sunday)
  • Crowd trend (spiking / stable / improving)
  • Weather condition (rain / clear)
  • Total report count + confidence level
  • Strict tone guide per badge level

PILLAR 2 — MOST INNOVATIVE
Jan Suraksha Bot

AI legal aid that actually files your complaint

Not a FAQ chatbot. This is a LangGraph-powered stateful agent that classifies railway crimes, retrieves relevant IPC sections via vector search, and auto-generates legally formatted FIR / CPGRAMS / RCT forms. In 3 languages. In under 3 LLM calls per conversation.

Architecture Manager Agent Pattern (LangGraph v3)

A central Manager node (pure Python, zero LLM) routes every turn. It eliminates phase tracking — routing is purely data-driven based on state flags. The graph is cyclic: Manager → {Classify | Execute | Respond} → Manager → END.

Smart Crime Classification

Single LLM call extracts: incident type (15+ categories), severity (low/medium/high/critical), user intent (file_complaint, know_rights, etc.), and structured entities (time, station, items lost, injuries, accused description).

incident: "theft" | severity: "high" | intent: "file_complaint"

Multilingual Entity Extraction

Works in English, Hindi, and Marathi (including code-mixed). Extracts station names validated against the actual WR/CR/HR station list — rejects vague locations like "inside train".

"Dadar pe phone chori" → {station: "Dadar", type: "theft"}

RAG Legal Knowledge Retrieval via Zilliz Vector Search

A curated legal corpus (IPC sections, Railways Act, case law, compensation schedules) is embedded with OpenAI text-embedding-3-small and stored in Zilliz Cloud. On each crime classification, the bot retrieves the top-5 relevant legal chunks, filtered by incident type and jurisdiction scope (Mumbai / National). The LLM never hallucinates legal sections — it only cites what RAG retrieves.

GRP FIR Format

Government Railway Police First Information Report. Pre-filled with incident details, IPC sections, complainant info. Ready to print and submit.

Authority: GRP (022-22694727)

CPGRAMS Portal

Centralized Public Grievance portal format. For non-criminal complaints like overcrowding, delays, staff misconduct, infrastructure issues.

Portal: pgportal.gov.in

RCT Compensation

Railway Claims Tribunal format for accidents, falls, platform gap incidents, deaths. Auto-calculates eligible compensation from legal RAG.

Min: Rs. 8,00,000 (death cases)

Zero LLM Rule-Based Authority Mapping

The bot doesn't guess which authority to contact. A deterministic mapping routes each incident type to the correct body: Theft/Assault → GRP | Sexual Harassment → Women Helpline 182 | Falls/Accidents → Railway Claims Tribunal | Corruption → Railway CVO | Delays/Overcrowding → Rail Madad. No LLM call needed — just a Python dictionary.

PILLAR 3 — ACTIONABLE INTELLIGENCE
Safety Intelligence Engine

Complaints become intelligence — for both passengers and police

Every complaint filed through Jan Suraksha doesn't just sit in a database. It feeds two real-time systems: passenger safety alerts and a police analytics dashboard.

Passengers Real-Time Safety Alerts

Every time a passenger opens the app, the engine scans the last 24 hours of complaints, identifies the top crime type, computes the peak danger window, and lists the affected stations. An LLM generates a 2-sentence actionable advisory like:

"5 theft reports in the last 24 hours, peaking between 5-7 PM at Dadar and Andheri. Keep your phone in your inner pocket and avoid standing near the door."

Police AI-Powered Analytics Dashboard

Six real-time analytics views, all computed from complaint data:

Complaints by Type

Horizontal bar chart showing incident distribution. Theft, assault, chain-snatching — all categorized with SVG icons.

Severity Breakdown

Color-coded bars: Critical (red), High (orange), Medium (amber), Low (blue). Helps prioritize response.

Station Hotspots

Top 6 stations ranked by incident count. Data-driven — never invented. Shows exactly where to deploy.

Hourly Crime Pattern

24-hour sparkline revealing peak crime windows. Usually 5-9 PM. Helps plan patrol shifts.

Weekly Trend

This week vs last week comparison. Shows if crime is rising or falling. Early warning system.

AI Patrol Recommendation

LLM-generated deployment suggestion: which stations, what time, what to watch for. Data-driven, never generic.

LLM Cost Optimization

Train search: 1 LLM call (reason generation, rule-based fallback available)
Bot chat: 2 calls (classify + respond)
Complaint filing: 3 calls (classify + FIR context + respond)
Legal RAG: Reuses context on re-entry — no duplicate embeddings
All critical paths: Rule-based fallbacks if LLM is unavailable

Zero-Dependency Auth

No JWT library. Tokens are HMAC-SHA256 signed JSON payloads with 24-hour expiry. Password hashing uses salted SHA-256 with 16-byte random salt. Zero external auth dependencies — just Python's stdlib hmac + hashlib.

Three Databases, Three Jobs

Neo4j AuraDB: Graph queries for train relationships + crowd report aggregation
SQLite (WAL mode): Users, complaints, auth — zero cloud dependency
Zilliz Cloud: 1536-dim legal embeddings for semantic retrieval

Graceful Degradation

If OpenAI is down: rule-based reason generation kicks in. If weather API fails: adaptive weights shift (hist × 0.60 + trend × 0.40). If Zilliz is unreachable: bot still classifies and routes to correct authority. Nothing crashes.
