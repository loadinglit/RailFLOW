Stage 1 — Graph Query

Smart train matching via graph traversal

Not a dumb table scan. The graph engine applies four filters simultaneously to return only relevant trains.

Line Detection
Automatically identifies WR, CR, or HR based on which stations appear in the search.
Direction Inference
Compares station indices in the ordered line map. Borivali(6) → Churchgate(26) = southbound. Only matching trains returned.
Time Window
Shows trains departing from 15 min before to 3 hours after search time. No 5am trains in an 8am search.
Route Coverage
Verifies the train actually covers your origin → destination segment. A Bandra-Churchgate train won't appear for Virar-Churchgate.
Stage 2 — Intelligence Engine

Three signals, weighted fusion

Each train is scored independently. No two trains get the same analysis — even on the same route.

50%

Historical Baseline

query: train_number + day_type + hour_bucket

Aggregates all crowd reports ever submitted for this specific train, on this type of day (weekday/saturday/sunday), during this hour slot. Produces a weighted score where each RED report = 100, YELLOW = 50, GREEN = 0.

Virar Fast 90015 | Monday | 08:00-09:00
56 reports: 40 RED, 12 YELLOW, 4 GREEN
score = (40/56 * 100) + (12/56 * 50) = 82
30%

Crowd Trend Detection

compare: last_7_days vs all_time_baseline

Detects if crowding is getting worse or better than usual. Computes the RED% in the last 7 days and subtracts the all-time RED%. A positive shift = worsening. Catches surges from festivals, exams, strikes, or cascading delays.

Baseline RED% = 61% | This week RED% = 82%
Shift = +21% → trend = "spiking" → score = 85

Baseline RED% = 61% | This week RED% = 55%
Shift = -6% → trend = "stable" → score = 30
20%

Live Weather Multiplier

source: OpenWeatherMap API — real-time Mumbai

Rain drives more commuters to trains (fewer autos/bikes). Heavy rain also causes platform rushing and slippery footboards. The weather signal acts as a crowd multiplier on top of the baseline.

Heavy rain (>10mm/hr) → score = 80
Light rain (2-10mm/hr) → score = 45
Clear / dry season → score = 0
Stage 3 — Score Fusion

Weighted sum → single badge

Fusion formula
Historical × 0.50 + Trend × 0.30 + Weather × 0.20 = Final Score
Example: (82 × 0.50) + (85 × 0.30) + (0 × 0.20) = 41 + 25.5 + 0 = 67 → YELLOW
GREEN0 – 39 — Safe
YELLOW40 – 69 — Caution
RED70 – 100 — Avoid
Stage 4 — LLM Reasoning

Natural-language explanation per train

The badge alone isn't enough. GPT-4o-mini generates a tone-matched one-liner that never contradicts the badge.

What the LLM receives

  • Badge: RED / YELLOW / GREEN
  • Train name and type
  • Day type (weekday / saturday / sunday)
  • Crowd trend (spiking / stable / improving)
  • Weather condition
  • Total report count
  • Strict tone guide per badge level

What the user sees

Virar Fast 08:18 RED
"Don't board this one — packed sardine-style since 7am."
Borivali Slow 08:32 GREEN
"Comfortable ride, you'll get a seat easily."
Borivali Fast 09:28 YELLOW
"Manageable if you skip first-class today."
The Flywheel

Community-powered intelligence

After boarding, users tap a crowd level. That report immediately feeds back into the engine. The next commuter sees a more accurate badge. This creates a self-improving loop.
